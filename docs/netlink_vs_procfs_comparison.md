# 采集方案对比：`setns + procfs` vs `persistent rtnetlink`

## 文档目的

本文档用于独立记录两种网络命名空间流量采集方案的测试结果与结论，便于后续评估采集主线方向，不与通用的命名空间采集说明混写。

## 对比方案

### 方案 A：`setns + procfs`

每轮采集时：

1. 在阻塞线程内进入目标网络命名空间
2. 读取 `/proc/self/task/<tid>/net/dev`
3. 解析当前业务所需的接口累计计数
4. 恢复原始网络命名空间

### 方案 B：`persistent rtnetlink`

每个命名空间预先建立持久化 `rtnetlink` client，采集时：

1. 复用已建立的 netlink 连接
2. 每轮发起 `RTM_GETLINK dump`
3. 接收并解析完整 link 消息流
4. 提取接口统计字段

## 对比口径

本次对比只统计**纯采集耗时**，不包含：

- 数据库存储
- 广播 / SSE
- HTTP 请求处理

并分别按以下两种口径验证：

- **生产并发口径**：按各自当前实现中的真实并发方式执行
- **全并发口径**：两种方案都按“全量命名空间同时采集”执行

## 测试环境

- 操作系统：Linux
- 内核版本：`3.10`
- 命名空间数量：约 `97`
- 接口数量：约 `394`

## 验证方法

本次结论基于以下手段综合得出：

- 纯采集 benchmark
- `strace` syscall 汇总
- `perf stat` CPU 总量统计
- `perf record/report` 热点分析
- 内核 `rtnetlink` 路径核对

## 结果摘要

| 对比口径 | `setns + procfs` | `persistent rtnetlink` | 结论 |
| --- | ---: | ---: | --- |
| 生产并发口径 | `1.933 ms / 轮` | `30.209 ms / 轮` | 原方案明显更快 |
| 全并发口径 | `1.912 ms / 轮` | `31.328 ms / 轮` | 原方案仍明显更快 |

## 关键观察

1. `persistent rtnetlink` 在当前环境下**没有体现出纯采集性能优势**
2. 即使将 `setns + procfs` 提升到全并发口径，结果仍明显快于 `persistent rtnetlink`
3. 差距主因不在 `setns`
4. 差距主因也不在 netlink client 初始化
5. 真正的开销集中在每轮 `RTM_GETLINK dump` 的总体成本

## 原因分析

### 1. `setns` 不是主瓶颈

验证结果表明，把原方案从生产并发提升到全并发后，平均每轮耗时几乎没有变化。  
这说明在当前环境里，`setns` 不是决定性成本来源。

### 2. 持久化 client 只省掉了连接初始化，没有省掉每轮 dump

`persistent rtnetlink` 的优势主要在于：

- 避免每轮重复创建 netlink 连接

但实际 steady-state 采集仍然需要：

- 每轮发起 `RTM_GETLINK dump`
- 读取完整响应流
- 解析完整 link 消息

因此，热路径上的主要成本仍然存在。

### 3. `Linux 3.10` 的 `rtnetlink` dump 路径存在全局锁竞争

在当前内核版本上，`RTM_GETLINK dump` 会进入 `rtnetlink` 控制路径，而该路径依赖全局 `rtnl_mutex`。  
这意味着：

- 即使不同命名空间有各自的 socket
- 即使不同命名空间各自维护独立 client
- 多个 namespace 并发发 dump 时，仍会在 `rtnetlink` 的全局控制平面上竞争同一把锁

这也是 `perf` 中出现明显锁竞争热点的重要原因。

### 4. kernel 返回的是完整 `ifinfo`，而不是当前业务只需要的少量计数器

当前业务真正需要的是：

- 接口名
- `rx_bytes`
- `tx_bytes`
- `rx_dropped`
- `tx_dropped`

而 `RTM_GETLINK dump` 走的是完整 link 信息路径。  
kernel 侧需要：

- 遍历设备
- 构造完整 `ifinfo` 消息
- 填充多类属性
- 分多包发送响应

因此，即使两边都可以视为“全量采集”，它们返回的“全量”形态并不相同：

- `procfs` 更接近业务所需的扁平快照
- `rtnetlink` 更接近完整 link 对象流

### 5. 用户态仍需承担协议请求/响应与属性解析成本

虽然业务代码表面上像是在直接消费结构化数据，但完整解析工作实际上发生在库内部。  
steady-state 过程中仍需承担：

- netlink 请求 / 响应
- 消息分发
- 属性解析
- 异步调度开销

这部分不是第一大头，但确实构成了额外成本。

### 6. 当前环境特征天然更有利于 `procfs`

本次环境的特征是：

- 命名空间数量较多
- 每个命名空间接口数较少
- 内核版本较老

在这种组合下：

- `/proc/net/dev` 文本非常短
- 读取和解析成本很低
- `rtnetlink dump` 的固定成本更难被摊薄

因此，`procfs` 更容易占优。

## 验证结论

综合 benchmark、syscall 分布、CPU 总量和热点分析后，可以得到以下判断：

- 在当前测试环境中，`persistent rtnetlink` 不适合作为默认的纯轮询采集热路径
- 当前差距主要来自：
  - `rtnetlink` 全局控制路径的锁竞争
  - kernel 构造完整 dump 消息的成本
  - netlink 协议层与用户态解析的附加开销
- 原方案 `setns + procfs` 仍然是当前环境下更合适的默认采集方式

## 当前决策

基于本次验证结果：

- 默认生产方向继续以 `setns + procfs` 为主
- `persistent rtnetlink` 暂不作为默认采集热路径推进
- 后续主线以 `main` 分支方向为准
- 迁移时可吸收 `netlink` 分支中与采集介质无关的通用优化，例如：
  - 写库与采集热路径解耦
  - 更稳健的命名空间运行时状态同步
  - 更清晰的生命周期与优雅关闭处理

## 后续重新评估的条件

在以下条件发生明显变化时，可以重新评估 `persistent rtnetlink` 方向：

- 部署环境升级到更新内核
- 单个命名空间的接口规模明显增大
- 采集模式从“每轮全量 dump”转向更窄的定点查询
- 热路径目标不再是简单的累计计数轮询

## 备注

本结论针对的是：

- 当前测试环境
- 当前代码实现
- 当前采集目标

它不直接代表所有内核版本、所有接口规模或所有采集模式下的绝对结论。
